\section{The pipeline, and reproducing Lawson's workflow}\label{sec:methods}

\subsection{The three-stage generation pipeline}
Each contig is carried through three stages. RFdiffusion3 generates \num{8} scaffold
backbones holding the epitope segment fixed at its contig position (one GPU task per contig,
\texttt{n\_batches=1}). Each backbone is stripped to its $C\alpha$/backbone trace and re-sequenced
by ProteinMPNN, which is told to hold the epitope residues \emph{fixed} --- so it designs only
the scaffold --- and draws \num{8} sequences per backbone (temperature \num{0.1}); the epitope's
fixed positions are read back from the contig, since an island sits at residues $N{+}1\ldots
N{+}\ell$ of an $N$-flank contig. AlphaFold3 then predicts each sequence single-sequence
(\texttt{--norun\_data\_pipeline}, one diffusion sample, seed~1), giving the
$\num{8}\times\num{8}=\num{64}$ designs per contig that the four filters score against the
designed structure. A small number of backbones fail to yield sequences in ProteinMPNN for
reasons we have not yet diagnosed, so the realized design count falls marginally short of the
nominal $\num{5568}\,V$ (for the $V=\num{20}$ run, \num{111322} of \num{111360}).
Generation code lives in \texttt{episcaf\_pipeline/} and \texttt{scripts/},
and the metrics, scoring, and visualization code in \texttt{episcaf\_analysis/}.

\paragraph{Benchmarking.} Rough per-step throughput, to size cluster wall-time requests:
RFdiffusion3 produces a contig's \num{8} backbones in under \SI{10}{\minute} on one A100;
ProteinMPNN sequences $\sim\!\num{215}$ backbones/hour ($\approx\SI{17}{\second}$ for a
backbone's \num{8} sequences); AlphaFold3 single-sequence predictions are fast but unmeasured
on our V100 nodes~[verify]. A run's wall-time then follows by multiplication --- e.g.\ a
\num{300}-backbone ProteinMPNN batch is $\approx\SI{1.4}{\hour}$, for which we request
\SI{8}{\hour} to leave margin.

\subsection{Reproducing Lawson's baseline}\label{sec:repro}
Before trusting any new number we re-implement Lawson's metrics and check them against his
ground truth. \texttt{compute\_metrics.py --validate} reads \texttt{dp2.parquet} (Lawson's
stored per-design values), recomputes \texttt{overall\_rmsd} and
\texttt{epitope\_chunk\_rmsd} from his ProteinMPNN PDBs and AlphaFold3 CIFs, and diffs them
against the stored values. Agreement on this diff is the precondition for everything
downstream: it confirms that our implementation of the four filters
(Section~\ref{sec:fourfilter}) matches the workflow we are extending, so that later
differences reflect the design protocol rather than a scoring bug.

\subsection{Code and data availability}\label{sec:availability}
The code is one git repository; the data and run outputs are not. Every figure and reported
number is regenerable by a named script from written-down inputs, and the repository records the
exact command for each. The generative stages (RFdiffusion3, ProteinMPNN, AlphaFold3) run as SLURM
array jobs on the TGen cluster, from the same repository checked out on the cluster's ephemeral
\texttt{/scratch}. The heavy outputs are then archived to a persistent workspace so they survive the
scratch purge:
\[
\texttt{\$WS} \;=\; \texttt{/tgen\_labs/altin/alphafold3/workspace/episcaf\_v2\_bneff}.
\]
Each shipped component maps to one run directory there. The two known-antibody scaffold arms are
\texttt{\$WS/runs/whole\_epitope\_rfd3} (C1, whole-epitope) and \texttt{\$WS/runs/dual\_island\_rfd3}
(C2, single-island); the polyclonal tiling arm is \texttt{\$WS/run\_12mer\_scaffolding} (C3); the
PfEMP1 arm is \texttt{\$WS/dp4\_8vdl} (8VDL). Each holds its per-design metrics table under its own
\texttt{05\_analysis} or \texttt{06\_score} subdirectory, which is the input every selection reads. The
three control components carry no run of their own: the linear tiled-30mers are built from the antigen
sequences, and the metric-space titration and the scaffold-disruption controls are both derived from the
C1 design pool. The oligo-encoding intermediates and the all-designs superset live alongside the runs
under \texttt{\$WS}; the shipped deliverables --- the assembled library, the encoder input, and the
verified synthesis order file --- are small and version-controlled in the repository itself, so they do
not depend on the cluster at all. The end-to-end run order and the exact per-stage commands are in
\texttt{docs/PIPELINE.md} and \texttt{docs/DP4\_LIBRARY.md}.
